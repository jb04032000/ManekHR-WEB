Indian Payroll Compliance Audit - Zari360 Salary Module

1. Executive Summary
   The current salary module is a solid operational payroll tool but is not yet compliant with Indian statutory requirements. It handles attendance-based pay, adjustments, payments, and CTC component breakdowns well. However, it has zero implementation of the major statutory deductions (PF, ESI, PT, TDS) and no compliance reporting (ECR, Form 24Q, Form 16). For any Indian employer with 10+ employees, several of these are legally mandated, making this a high regulatory risk product as-is. The architecture (component templates, PayrollConfig toggles) is well-positioned to absorb these additions without a rewrite.

4. Detailed Findings on Critical Gaps
   A. Statutory Deduction Engine - The Biggest Gap
   The module has a component template system that can model earnings components (Basic, HRA, etc.) but has no concept of statutory deductions with their own calculation rules. PF, ESI, and PT aren't just free-form adjustments - they have applicability rules (thresholds, state, establishment size), fixed formulas, and employer-side cost implications. You need a StatutoryConfig on the workspace (PF establishment code, ESI code, applicable state for PT, etc.) and a dedicated statutory calculation step in buildSalaryRecordData().
   B. TDS is the Most Complex Gap
   TDS on salary requires: (1) employee declares investments at year start, (2) system projects annual taxable income monthly, (3) divides remaining tax liability by remaining months. Without an investment declaration workflow and a tax slab engine (old vs new regime), TDS cannot be computed correctly. This is also what drives Form 16.
   C. PF ECR is a Hard Regulatory Requirement
   Any establishment with 20+ employees must file PF ECR monthly with EPFO. The file format is fixed-width text, and it includes UAN numbers, wages, contributions. This requires storing UAN on TeamMember and generating the file from payroll data. This is a blocker for enterprise sales.
   D. Bonus Act Partial Implementation Risk
   The bonus adjustment category exists but is entirely manual. If a user records a bonus adjustment without respecting the Bonus Act (eligibility в‰¤ в‚№21,000, min 8.33%, max 20%, calculated on в‚№7,000 or minimum wage whichever higher), they're non-compliant. The UI should either guide or enforce this, especially in the professional/enterprise preset.
   E. PT - Frequently Underestimated
   Professional Tax slabs vary by state (Maharashtra, Karnataka, West Bengal, etc. have different slabs and frequencies). This needs a state field on the workspace or branch, and a PT slab configuration system.
